2. Introduction

2.1 What is a Web Application Vulnerability Assessment?

A web application vulnerability assessment is a structured security review process that identifies, classifies, and prioritizes security weaknesses in a web application. Unlike a penetration test, which actively exploits vulnerabilities, a vulnerability assessment focuses on discovering and documenting weaknesses so they can be remediated before a real attacker finds them.

2.2 Why is this important?

Web applications are among the most common targets for cyberattacks. According to OWASP (Open Web Application Security Project), vulnerabilities like SQL Injection and Cross-Site Scripting consistently rank among the most dangerous and most exploited security flaws worldwide. Identifying these weaknesses early helps organizations:

  • Protect sensitive user data from theft or exposure
  • Prevent unauthorized access to systems and databases
  • Comply with data protection regulations such as GDPR
  • Maintain application integrity and user trust

2.3 About the Target Application

testasp.vulnweb.com is a test web application intentionally built with security vulnerabilities by Acunetix, a cybersecurity company. It is provided specifically for security professionals, students, and researchers to practice vulnerability assessment legally and ethically. It simulates a real-world ASP-based web application with a Microsoft SQL Server database backend.

2.4 About OWASP ZAP

OWASP ZAP (Zed Attack Proxy) is a free, open-source web application security scanner maintained by the Open Web Application Security Project. It is one of the most widely used tools for identifying vulnerabilities in web applications. ZAP works by:

  • Acting as a proxy between the browser and the target application to intercept traffic
  • Spidering the application to discover all pages, forms, and endpoints
  • Running an active scan that probes each discovered page for known vulnerabilities
  • Generating a detailed alerts report categorized by severity

4. Methodology

The assessment followed a structured four-phase approach based on industry-standard security testing practices:

Phase 1 — Setup and Configuration

  • Installed and launched OWASP ZAP v2.17.0 on macOS
  • Configured browser proxy to route all traffic through ZAP at localhost:8080
  • Confirmed ZAP was successfully intercepting traffic from the target
  • Set target URL to http://testasp.vulnweb.com

Phase 2 — Reconnaissance (Spidering)

  • Used ZAP's Traditional Spider to crawl the application and map all pages, links, and forms
  • Used ZAP's Ajax Spider with Firefox to discover dynamically loaded content
  • Result: A complete site map of all accessible endpoints was generated

Phase 3 — Active Scanning

  • Launched ZAP's Active Scan against the full discovered site map
  • ZAP probed each page and parameter for known vulnerability patterns including SQL Injection, XSS, Path Traversal, CSRF, and insecure headers
  • Both GET and POST parameters were tested
  • Result: 19 alerts generated across 4 severity levels

Phase 4 — Analysis and Reporting

  • Reviewed each alert in ZAP's Alerts panel and verified findings
  • Categorized findings by severity (High, Medium, Low, Informational)
  • Documented each vulnerability with description, impact, and remediation advice
  • Exported the full ZAP HTML scan report
  • Captured screenshot evidence of the alerts panel

6.1 SQL Injection

Instances: 4
OWASP Category: A03:2021 – Injection
What is it?

SQL Injection occurs when an attacker inserts malicious SQL code into an input field such as a login form or search box. If the application does not properly validate or sanitize that input, the malicious code gets executed directly by the database. This is one of the most dangerous and most exploited vulnerabilities in existence.

What ZAP found:

ZAP detected 4 endpoints where user input is passed directly into SQL queries without sanitization. Entering a payload such as ' OR '1'='1 into a login field could bypass authentication entirely.

Impact:
  • Complete bypass of login authentication
  • Unauthorized access to the entire database
  • Theft, modification, or deletion of sensitive records
  • Possible remote code execution on the server
Recommendation:
Use parameterized queries (prepared statements) instead of building SQL by concatenating user input. Apply the principle of least privilege to database accounts. Consider using an ORM such as Entity Framework or Hibernate.
High

6.2 SQL Injection – MsSQL Time Based

Instances: 2
OWASP Category: A03:2021 – Injection
What is it?

A blind SQL injection technique where the attacker cannot see the output directly. Instead, they inject commands that cause the database to pause for a set number of seconds. If the response is delayed, the injection succeeded. This confirms the backend uses Microsoft SQL Server.

Impact:

Silent data exfiltration without visible output. An attacker can extract the full database schema, usernames, passwords, and sensitive records over time — all without triggering obvious errors.

Recommendation:
Same as 6.1 — parameterized queries and input validation. Additionally, monitor database query execution times for anomalies and implement a Web Application Firewall (WAF).
High

6.3 External Redirect

Instances: 2
OWASP Category: A01:2021 – Broken Access Control
What is it?

The application accepts a URL as a parameter and redirects users to it without validating whether it is safe or trusted. For example: http://testasp.vulnweb.com/redirect?url=http://malicious.com

Impact:
  • Phishing attacks — users trust the original domain and get redirected to fake login pages
  • Malware distribution via redirects to infected sites
  • Reputational damage to the organization
Recommendation:
Never use user-supplied input directly in redirect URLs. Maintain a server-side whitelist of allowed destinations. Use indirect references (e.g. ?dest=1 mapped server-side) instead of passing raw URLs.
High

6.4 Path Traversal

Instances: 2
OWASP Category: A01:2021 – Broken Access Control
What is it?

Path traversal allows an attacker to access files outside the intended web root by injecting sequences like ../../ into file path parameters, navigating up the directory tree to read sensitive system files.

Impact:
  • Access to server configuration files and credentials
  • Exposure of application source code and API keys
  • Full server compromise in severe cases
Recommendation:
Validate and sanitize all file path inputs. Use a whitelist of allowed file names. Apply server-side path canonicalization before use. Run the application with minimum required file system permissions.
Medium

6.5 Absence of Anti-CSRF Tokens

Instances: 4
OWASP Category: A01:2021 – Broken Access Control
What is it?

Cross-Site Request Forgery (CSRF) tricks a logged-in user's browser into making unwanted requests to a trusted site. Without CSRF tokens, the application cannot distinguish legitimate requests from forged ones sent by a malicious third-party site.

Impact:

Unauthorized actions on behalf of authenticated users — including password changes, data modifications, or financial transactions — without the user's knowledge or consent.

Recommendation:
Implement the synchronizer token pattern — include a unique secret CSRF token in every state-changing form. Use the SameSite cookie attribute and validate Origin/Referer headers on sensitive requests.
Medium

6.6 Content Security Policy (CSP) Header Not Set

What is it?

A Content Security Policy is an HTTP header that tells the browser which content sources are trusted. Without it, the browser executes any script it encounters, making XSS attacks significantly more effective and damaging.

Recommendation:
Add Content-Security-Policy: default-src 'self' to all HTTP responses. Gradually allow trusted third-party sources as needed.
Medium

6.7 Application Error Disclosure

What is it?

The application displays detailed error messages in the browser, including stack traces, database query details, server file paths, and software versions. This information is a goldmine for attackers mapping the application's internals.

Recommendation:
Show only generic, user-friendly error messages to end users. Log full error details server-side only. Disable debug mode in all production environments.
Medium

6.8 Missing Anti-Clickjacking Header

What is it?

Clickjacking embeds the target application in a hidden iframe on a malicious page, tricking users into clicking on invisible elements. Without X-Frame-Options, any website can embed this application.

Recommendation:
Add the HTTP header: X-Frame-Options: DENY or use CSP: Content-Security-Policy: frame-ancestors 'none'
Low

6.9 Cookie Security Issues

Findings: Cookie No HttpOnly Flag | Cookie without SameSite Attribute
What is it?

Session cookies missing the HttpOnly flag can be read by JavaScript, enabling session theft via XSS. Cookies missing the SameSite attribute are sent with cross-site requests, enabling CSRF attacks.

Recommendation:
Set HttpOnly, SameSite=Strict, and Secure flags on all session cookies: Set-Cookie: session=abc; HttpOnly; SameSite=Strict; Secure
Low

6.10 Information Disclosure via HTTP Headers

Findings: X-Powered-By Header | Server Version Header | X-Content-Type-Options Missing | Debug Error Messages
What is it?

The server is leaking technical details through HTTP response headers — including the web framework (e.g. X-Powered-By: ASP.NET) and server version (e.g. Server: Microsoft-IIS/7.5). This helps attackers identify known exploits for those specific versions.

Recommendation:
Remove or suppress X-Powered-By and Server headers in the web server configuration. Add X-Content-Type-Options: nosniff to prevent MIME-type sniffing attacks.

8. Recommendations

🔴 Immediate Actions (High Priority)

  1. Fix SQL Injection — Replace all dynamic SQL queries with parameterized queries or prepared statements. This is the single most important fix.
  2. Fix Path Traversal — Validate and sanitize all file path inputs using server-side path whitelisting.
  3. Fix External Redirect — Remove or restrict URL redirect parameters. Use a server-side whitelist of allowed destinations only.

🟠 Short-Term Actions (Medium Priority)

  1. Implement CSRF tokens — Add unique tokens to all state-changing forms across the application.
  2. Set security headers — Add CSP, X-Frame-Options, and X-Content-Type-Options to all HTTP responses.
  3. Disable error disclosure — Show only generic error messages to users; log details server-side only.

🟡 Ongoing Actions (Low Priority)

  1. Secure all cookies — Add HttpOnly, SameSite, and Secure flags to every session cookie.
  2. Remove leaking headers — Suppress X-Powered-By and Server version headers from all responses.
  3. Clean HTML source — Remove all developer comments from production code.
  4. Enforce HTTPS — Redirect all HTTP traffic to HTTPS across the entire application.

9. Conclusion

This vulnerability assessment of testasp.vulnweb.com identified 19 security vulnerabilities, including 4 of high severity. The application demonstrates several critical weaknesses commonly found in real-world web applications — particularly around input validation (SQL Injection, Path Traversal) and access control (External Redirect, CSRF).

The use of OWASP ZAP proved highly effective in automatically discovering a wide range of vulnerabilities in a short timeframe. However, automated tools should always be complemented with manual testing to uncover logic flaws and context-specific vulnerabilities that scanners may miss.

Addressing the high-severity findings — especially SQL Injection — should be treated as an urgent priority. These vulnerabilities, if exploited by a malicious actor, could lead to complete application and database compromise, data theft, and significant reputational damage.
